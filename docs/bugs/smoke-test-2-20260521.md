## Bug #1 — CRITICAL: Zone Transition Event Name Mismatch

### Summary
When a player steps on an exit tile, the server fires a `zone_change` event. The Python SDK only auto-switches its Phoenix topic on `zone_entered`. The topic mismatch means all subsequent actions are sent to the wrong (old) zone — the player appears to never move.

### Root Cause
**Server** (`lib/agent_mmo/world/zone_ticker.ex:572`):
```elixir
send_player_event(player_id, %{
  type: "zone_change",     # ← "zone_change"
  zone_id: dest_zone,      # ← key: "zone_id"
  position: %{x: dest_x, y: dest_y}
})
```

**Channel** (`lib/agent_mmo_web/channels/game_channel.ex:173-180`):
```elixir
def handle_info({:player_event, %{type: "zone_change"} = payload}, socket) do
  # Correctly updates socket's zone subscription and pushes to client as "event"
  push(socket, "event", payload)
end
```

**SDK** (`clients/python/tavernbench/client.py:414-416`):
```python
if event == "event":
    etype = payload.get("type", "")
    if etype == "zone_entered":      # ← looks for "zone_entered"
        self._topic = f"zone:{payload.get('to_zone', ...)}"  # ← uses "to_zone" key
```

Three mismatches:
1. Server event type is `"zone_change"` but SDK listens for `"zone_entered"`
2. Server key is `"zone_id"` but SDK reads `"to_zone"`
3. After zone transition, client needs to **leave and re-join** the new zone channel to receive ticks from it (the channel server handles PubSub re-subscription via `handle_info({:player_event, %{type: "zone_change"}}...)` but the client side never calls `join()` again on the new zone topic)

### Impact
Complete: after any zone transition via movement, the player is invisible to the client. All subsequent actions (move, speak, attack, pickup) are enqueued to the old zone's ticker which no longer has the player entity. The playthrough cannot progress past the first zone.

### Fix
**Option A (server-side):** Change the event type and key in `send_player_event` calls in `check_exit_transition` and `handle_enter` to match what the SDK expects:
```elixir
send_player_event(player_id, %{
  type: "zone_entered",   # match SDK
  from_zone: state.zone_id,
  to_zone: dest_zone,     # match SDK key
  position: %{x: dest_x, y: dest_y}
})
```
Apply same change to `handle_enter/5` at line ~212.

**Option B (SDK-side):** Update SDK to handle both names and key:
```python
if etype in ("zone_entered", "zone_change"):
    new_zone = payload.get("to_zone") or payload.get("zone_id", "")
    self._topic = f"zone:{new_zone}"
```

**Recommendation:** Do both for consistency with the protocol spec (section 3.2 / zone_entered). The SDK needs the topic switch; the server's naming should match the protocol docs.

Additionally, the client must re-join the new zone channel after topic switch so it receives ticks. Current `_dispatch` updates `self._topic` which fixes action routing, but the client still only gets ticks from the original WS join. This works if the channel server correctly re-subscribes to PubSub (it does via `handle_info zone_change`), so the fix is just the naming alignment.

### Files to change
- `lib/agent_mmo/world/zone_ticker.ex` lines 572-575 and ~212-216
- `clients/python/tavernbench/client.py` lines 414-416

## Bug #2 — HIGH: Spectator Channel Not Implemented

### Summary
The protocol spec (section 7, `docs/protocol.md`) defines a `spectate:<zone_id>` channel. No server-side channel module exists for it. Attempting to join `spectate:tavern` returns a join error.

### Evidence
```
grep -rn "spectate" lib/  → 0 matches
```

The `user_socket.ex` only routes `zone:*` topics to `GameChannel`. There is no `SpectateChannel`.

### Impact
REQ 5 (spectator channel receives state broadcasts) cannot pass. The Go TUI demo client also won't work.

### Fix
Implement `AgentMmoWeb.SpectateChannel` that:
1. Joins `spectate:<zone_id>` topics
2. Subscribes to `Phoenix.PubSub` on `"zone:<zone_id>"` (same topic as players)
3. Forwards `tick_broadcast` messages as `tick` pushes
4. Rejects any action messages (`handle_in` for unknown events → error)
5. Register in `user_socket.ex`: `channel "spectate:*", AgentMmoWeb.SpectateChannel`

No auth bypass needed — api_key check is already on the socket level.

### Files to create/change
- `lib/agent_mmo_web/channels/spectate_channel.ex` (new)
- `lib/agent_mmo_web/channels/user_socket.ex` (add channel route)

## Bug #3 — MEDIUM: Wrong-Path Penalty Not Applied on Zone Entry

### Summary
The scenario YAML marks `exit_southwest` in `village_square` as `wrong_path: true` with `wrong_path_penalty: 10`. Entering `old_mill` should immediately deduct 10 from the player's score. This deduction never happens.

### Root Cause
The `exits_meta` list stored in ETS zone_meta only carries `id, position, label, destination_zone, destination_position` (ScenarioLoader line 129-139). The `wrong_path` and `wrong_path_penalty` fields are never parsed or stored.

`check_exit_transition` in ZoneTicker has no code to apply a penalty when transitioning to a wrong-path zone.

### Evidence
In `lib/agent_mmo/world/scenario_loader.ex` exits_meta construction (lines 128-140):
```elixir
%{
  id: e["id"],
  position: pos,
  label: e["label"] || "",
  destination_zone: e["destination_zone"],
  destination_position: dest_pos
  # wrong_path and wrong_path_penalty are NOT here
}
```

### Impact
Agents that take the red-herring path to `old_mill` receive no score penalty. The benchmark score for wrong-path reasoning is inaccurate (inflated by 10 points).

### Fix
1. In `parse_exit` (or inline in `seed_zone`), capture `wrong_path` and `wrong_path_penalty`:
```elixir
%{
  id: e["id"],
  position: pos,
  label: e["label"] || "",
  destination_zone: e["destination_zone"],
  destination_position: dest_pos,
  wrong_path: e["wrong_path"] || false,
  wrong_path_penalty: e["wrong_path_penalty"] || 0
}
```
2. In `check_exit_transition` (zone_ticker.ex ~line 556), after matching exit:
```elixir
if matching_exit[:wrong_path_penalty] && matching_exit[:wrong_path_penalty] > 0 do
  PlayerSession.deduct_score(player_id, matching_exit[:wrong_path_penalty], "wrong_path:#{dest_zone}")
end
```

### Files to change
- `lib/agent_mmo/world/scenario_loader.ex` (exits_meta parsing)
- `lib/agent_mmo/world/zone_ticker.ex` (check_exit_transition, apply penalty)
